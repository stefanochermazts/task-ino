## Deep Story Analysis – Patterns Extracted

### What went well

1. **Non-blocking fire-and-forget applied correctly throughout**  
   All event log writes used `.catch(console.error)` pattern, never blocking the planning mutation hot path. No deviations found across 7 stories.

2. **Fake IDB test pattern reused without modification**  
   The pattern established in `e2eeClientCrypto.test.js` was reused cleanly in `eventLogStore.test.js` with no adaptation cost, validating the pattern's extractability.

3. **`validateTaskFields` extracted and reused (DRY achieved)**  
   Story 4.2 identified duplication between export and integrity validation and extracted a shared utility. No subsequent integrity finding touched this concern.

4. **Feedback channel isolation held across the full epic**  
   `#timeline-feedback` and `#recovery-feedback` remained independent from `#export-feedback` and `#control-feedback`. The Epic 3 lesson was fully applied.

5. **Adversarial review continued to catch real defects**  
   Every story had at least one HIGH finding that would have reached production without the review cycle. The discipline proved its value at scale across a 7-story epic.

6. **Error sanitization enforced in all new UI paths**  
   Recovery flow, timeline, and all destructive path error messages were user-safe. No raw technical strings surfaced in user-facing copy.

### What did not go well

1. **Every story had at least one HIGH finding (7/7)**  
   This is the most significant signal of the epic. While the review cycle caught the defects, the pattern suggests a systemic gap between implementation and acceptance criteria validation at the story level. Notable findings:
   - S-4.1: Store name contract mismatch (`events` vs `planningEventLog`)
   - S-4.2: `DAY_CYCLE_CONTINUITY` accepted regex-valid but unparseable dates
   - S-4.3: Non-deterministic replay (wall-clock dependency in `rescheduled` logic) + missing rebuild trigger after delete
   - S-4.4: Timeline did not auto-load on panel open (AC1 violated)
   - S-4.5: `REBUILD_PARTIAL` metadata lost through `executeRecoveryAction`
   - S-4.6: `deleteSyncedPlanningData` reported false-positive success on unconfirmed remote response
   - S-4.7: Runtime invariant guard neutralized by generic `catch` block

2. **`initializePlanningInboxApp.js` is accumulating complexity**  
   This single file was meaningfully modified in 5 of 7 stories (4.3, 4.4, 4.5, 4.6 indirectly, 4.7). It handles timeline wiring, recovery wiring, export wiring, control panel wiring, and projection rebuild triggers. Without decomposition, this file becomes a maintenance liability for Epic 5's UI-heavy stories.

3. **Partial success metadata propagation pattern repeated twice**  
   `REBUILD_PARTIAL` (S-4.5) and `LOCAL_DELETE_E2EE_PARTIAL` (S-4.7) both followed the same pattern: the inner command returned a meaningful code, but the outer `executeRecoveryAction` wrapper silently discarded it with a generic `{ ok: true }`. Same bug, same root cause, two different stories.

4. **Story file list tracking still produced one gap**  
   Despite being an Epic 3 action item and a MEDIUM finding in S-4.1, `deleteLocalPlanningData.test.js` (modified for `onAfterDelete` callback tests in Story 4.3) was absent from Story 4.3's File List. Found and corrected during this retrospective.

## Key Insights

1. **Invariant guards must be outside generic catch blocks**  
   The most dangerous failure mode in Epic 4 was an invariant assertion that could not surface because it was caught and downgraded to a generic error. Invariants must re-throw or use a channel that bypasses generic error swallowing. This applies to any future `assertX()` function placed inside orchestrator `try/catch` blocks.

2. **Partial success is a first-class result, not an edge case**  
   Every time a command chain wraps an inner result, the wrapper must explicitly propagate `code` and `message` from the inner result — not just `ok`. The pattern `return { ok: true }` from a wrapper is almost always wrong when the inner result contains richer metadata.

3. **Auto-load on open is a non-negotiable for panel-based UI**  
   Any collapsible panel that displays data should auto-load when opened. Requiring the user to click a refresh button after opening a panel violates the principle of least surprise. This pattern should be part of the UI development baseline for Epic 5 panels.

4. **`initializePlanningInboxApp.js` needs decomposition before Epic 5**  
   Epic 5 is UI/UX-heavy (accessibility, responsive layout, keyboard operability). Adding more wiring to this file without decomposition will make reviews, testing, and onboarding harder. Splitting into domain-focused UI sub-modules is now critical prep, not optional refactoring.

5. **Contract-tested determinism catches entire classes of bugs**  
   The snapshot-reference-date pattern in `replayEventLogPatch` replaced wall-clock dependency and made tests time-independent. This is the correct model for any function that applies historical events to a reference context.

## Retrospective Closure

Epic 4 met its objective: recovery and supportability tooling was introduced without making destructive reset the default path, without coupling the event log to runtime state, and without compromising the invariant-safe command architecture. The most important outcome was the consistent enforcement of non-destructive defaults at the command layer — not as a UI convention but as a tested, asserted contract.

The most important process signal from this epic is that **every story had a HIGH finding**. This is not a failure of the review process — it is a success of it. But it also indicates that story-level self-verification (against ACs, not just against test pass/fail) could be tightened to catch more defects before review, reducing fix-cycle overhead.

The creation of the `bmad-story-file-list-sync.mdc` cursor rule during this retrospective closes the last open action item from Epic 3 with a permanent structural fix.
