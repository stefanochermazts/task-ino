## Deep Story Analysis – Patterns Extracted

### What went well (themes)

1. **Invariant pipeline scaled cleanly**
   - `mutatePlanningState` remained the single write-path as we added new actions (`setTaskArea`, reschedule single/bulk, continuity, explicit retain).

2. **Strong separation of concerns held under growth**
   - Structural actions (area / Today membership) stayed independent from temporal actions (`scheduledFor`, `retainedFor`).

3. **Atomicity was treated as a first-class property**
   - Bulk reschedule built a rollback-capable persistence path with tests that simulate mid-transaction write failures.

4. **Review discipline prevented “silent drift”**
   - Code review surfaced meaningful semantic gaps (e.g., “explicitly retained” meaning) and correctness issues (UTC vs local day boundary).

### What didn’t go well (themes)

1. **Date semantics were not explicit enough early**
   - The day-boundary / “today” calculation surfaced a subtle but product-critical question: “today” should follow **user local day**, not UTC.

2. **Specification language (“retained”) needed an explicit data model**
   - AC wording included “explicitly retained”, but without an explicit persisted marker it would have remained ambiguous and fragile.

3. **Mock timing / transactional edge cases are easy to underestimate**
   - The atomic rollback test harness needed enhancements to reliably simulate failure modes and avoid false positives.

---

## Key Insights

1. **Temporal correctness is a domain invariant**
   - “Today” is not just a UI label; it must be defined and implemented consistently (local-day semantics) anywhere continuity and scheduling are enforced.

2. **Ambiguous AC verbs require explicit persisted state**
   - If an AC says “explicitly retained”, it must map to a stored, queryable marker (not an implicit behavior).

3. **Rollback testing must simulate failure mid-flight**
   - Atomicity claims are only trustworthy when tests prove behavior under partial write failures.

4. **Bootstrap actions are part of the domain**
   - Startup flows (continuity enforcement before projection render) need the same rigor and test coverage as button-driven actions.

## Next Epic Preview: Epic 3 – Trust Controls, Sync, and Data Ownership

Epic 3 adds optional sync, mode management, conflict resolution, transparency, and export/delete controls.

**Dependencies on Epic 2:**

- Stable invariant layer (`mutatePlanningState`) and domain error vocabulary
- Clear separation between structural vs temporal semantics
- Day-cycle continuity rules (must remain isolated from sync variability)

**Key risks if we’re not careful:**

- Sync engine accidentally mutates planning state “because server said so”
- Conflict resolution introduces implicit carry-over or breaks determinism
- Date/time semantics drift across devices without a canonical definition
